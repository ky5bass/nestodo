# ガイドライン詳細

本ドキュメントは、Skill Creator Skillが生成するSkillの品質を確保するためのガイドラインを記述する。

## Progressive Disclosure原則の適用

Skillファイルの設計において、Progressive Disclosure原則を適用し、情報の読み込みを段階的に制御する。

### SKILL.md本文の制約

- SKILL.md本文は500行以下に抑える
- 500行を超える場合は、詳細な内容をreferences/配下のファイルに分離する
- SKILL.md本文には要約のみを残し、詳細は参照先に委ねる

### frontmatter descriptionの設計

- descriptionはSkillのアクティベーション時に読み込まれるため、簡潔かつ情報量の多い記述にする
- 機能、使用タイミング、トリガーキーワードの3要素を必ず含める
- 1024文字以内で最大限の情報を伝える

### 関連ファイルの配置戦略

- scripts/、references/、assets/は必要時のみ読み込まれるため、大きなコンテンツも配置可能
- 各ファイルは1トピック1ファイルの原則で焦点を絞る
- ファイル参照は相対パスで1階層以内に制限する

## ベストプラクティスへの準拠

`agent-skills-best-practices.md`に記載された全ルールに準拠し、高品質なSkillを生成する。

### 段階的な改善アプローチ

- 最初から完璧なSkillを目指すのではなく、評価から始めて段階的に改善する
- ユーザーのフィードバックを取り入れ、ワークフローやルールを調整する
- バリデーション結果に基づき、具体的な改善提案を行う

### エージェントの視点での設計

- Skillの指示はエージェント（AI）が実行することを前提に記述する
- 曖昧な表現を避け、具体的で実行可能な指示を記述する
- 実際のシナリオでの使用を想定し、エッジケースも考慮する

### 品質チェックの実施

- 生成後にバリデーションチェックリストを実行する
- name形式、description品質、行数制約、ファイル参照、セクション構成を検証する
- 違反が検出された場合は修正提案を提示する

## エラーハンドリング

Skill作成プロセス中に発生しうるエラーに対して、適切に対処する。

### Web検索失敗時

- ユーザーに手動での情報提供を依頼する
- ヒアリングを中断せず、代替手段で情報収集を継続する
- 「Web検索で情報を取得できませんでした。{不足情報}について手動で教えていただけますか？」のように具体的に依頼する

### ファイル生成エラー時

- エラー内容をユーザーに通知する
- 再試行の選択肢を提示する（再試行 / 別の方法で生成 / 中断）
- 部分的に生成済みのファイルがある場合は、その状態を明示する
- 「ファイル生成中にエラーが発生しました: {エラー内容}。再試行しますか？」のように通知する

### 修正依頼時

- ユーザーからの修正依頼を受けた場合、まず修正案を提示する
- 修正案にはファイル名、変更箇所、変更前後の内容を含める
- ユーザーの明示的な承認を得てから修正を実行する
- 承認なしでの修正実行は禁止する
