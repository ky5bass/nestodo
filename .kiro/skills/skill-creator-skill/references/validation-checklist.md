# バリデーションチェックリスト

生成されたSkillがベストプラクティスに準拠しているかを検証するためのチェックリストである。Skill生成完了後（ワークフロー ステップ10）で全項目を順番に検証する。

## チェック項目一覧

| # | チェック項目 | 基準 | 違反時の対応 |
|---|---|---|---|
| 1 | name形式 | kebab-case形式（正規表現: `^[a-z0-9]+(-[a-z0-9]+)*$`） | 自動変換候補を提示 |
| 2 | name文字数 | 64文字以内 | 切り詰め候補を提示 |
| 3 | nameとディレクトリ名の一致 | frontmatterのnameと配置先ディレクトリ名が完全一致 | 修正を求める |
| 4 | description - 機能 | Skillの機能が具体的に記述されている | 修正案を提示 |
| 5 | description - 使用タイミング | いつ使うべきかが明示されている | 修正案を提示 |
| 6 | description - トリガーキーワード | アクティベーション用キーワードが含まれている | 修正案を提示 |
| 7 | description文字数 | 1024文字以内 | 短縮案を提示 |
| 8 | SKILL.md行数 | 500行以下 | references/への分割を実行 |
| 9 | ファイル参照パス | 相対パスで記述されている | 修正案を提示 |
| 10 | ファイル参照の深さ | 1階層以内（例: `references/guide.md` は可、`references/sub/guide.md` は不可） | 修正案を提示 |
| 11 | セクション構成 - 概要 | 概要セクション（`## 概要`）が存在する | セクションを追加 |
| 12 | セクション構成 - ワークフロー | ワークフローセクション（`## ワークフロー`）が存在する | セクションを追加 |
| 13 | セクション構成 - ルール | ルールセクション（`## ルール`）が存在する | セクションを追加 |

## バリデーション実行手順

1. 上記チェック項目を#1から#13まで順番に検証する
2. 違反が検出された場合、違反箇所と修正提案をユーザーに提示する
3. ユーザーの承認を得てから修正を実行する（確認なしでの自動修正は禁止）
4. すべてのチェックに合格した場合、生成された成果物の一覧をユーザーに提示する

## 検証結果の出力形式

### 全チェック合格時

```
【バリデーション結果】
✅ すべてのチェックに合格しました

📁 {配置先パス}
  ├── SKILL.md          - Skill定義ファイル（{行数}行）
  ├── scripts/          - （生成した場合のみ表示）
  ├── references/       - （生成した場合のみ表示）
  └── assets/           - （生成した場合のみ表示）
```

### 違反検出時

```
【バリデーション結果】
⚠️ {N}件の違反が検出されました

❌ チェック#{番号} {チェック項目}: {違反内容}
   修正案: {具体的な修正提案}

この修正を実行してよろしいですか？
```
