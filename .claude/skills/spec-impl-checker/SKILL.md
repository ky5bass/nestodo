# Spec と実装の整合性レビュー

Kiro が spec を生成し Codex が実装した後、実装 PR を人間がレビューする前に使う。
このスキルは開発フローのステップ6（実装 PR のコード品質・spec 整合性チェック、およびレビュー指摘対応の再確認）を担い、spec に対して実装差分が過不足なく対応しているかを確認する。

## モード判定

- 未レビューの PR、ローカル差分、または通常のレビュー依頼では「初回レビューモード」を実行する
- 「レビュー指摘対応を再確認」と依頼された場合、または同じ PR にこのスキルの識別子付き指摘がある場合は「指摘対応再確認モード」を実行する

## 呼び出しトリガー

次のような依頼ではこのスキルを使う。

- `/spec-impl-checker コード PR #XX をレビューしてください`
- 「この実装 PR が spec を満たしているか確認して」
- 「Codex の実装差分と `.kiro/specs/<spec名>/` の整合性を見て」
- 「spec 完了処理の前に実装の過不足を確認して」

対象 spec が明示されていない場合は、PR の説明、変更ファイル、ブランチ名、または差分から関連する `.kiro/specs/<spec名>/` を推定する。
推定できない場合は、レビュー結果の冒頭で「対象 spec 不明」と明記し、確認できた範囲だけを報告する。

## 入力として扱う情報

依頼文に含まれる情報を優先して使う。

- PR URL または PR 番号
- 対象 spec のパス（例: `.kiro/specs/daily-export/`）
- 比較対象ブランチ
- ローカル差分

PR 番号または URL がある場合は、`gh` CLI で PR 本文、変更ファイル、差分を確認する。
ローカル作業ツリーのレビューを依頼された場合は、`git status` と `git diff` を確認する。
GitHub MCP ツールは使わず、必要な GitHub 操作は `/github-operations` を参照する。

## 読むファイルと順番

1. `docs/CONTRIBUTING.md`
   - ステップ6の責務、ステップ7の人間による動作確認、ステップ8で Kiro が行う spec 完了処理との境界を確認する。
2. `.kiro/steering/development-rules.md`
   - spec 運用、言語、tasks.md 削除ルール、GitHub 操作ルールを確認する。
3. 対象 spec の `.kiro/specs/<spec名>/requirements.md`
   - ユーザーストーリーと受け入れ基準を抽出する。
4. 対象 spec の `.kiro/specs/<spec名>/design.md`
   - 設計判断、インターフェース、コマンド、データ構造、制約を抽出する。
5. 対象 spec の `.kiro/specs/<spec名>/tasks.md`
   - 存在する場合は、各タスクの実装・テスト完了状況を確認する。
   - 実装 PR の段階では存在してよい。実装 PR マージ後のステップ8で Kiro が削除する。
6. PR またはローカルの実装差分
   - 変更ファイル一覧を先に見てから、spec に関係するコード、テスト、設定、ドキュメントの差分を読む。
7. 必要に応じて `docs/requirements-spec.md`
   - 全体アーキテクチャ、技術スタック、共通データモデルとの整合性が問題になりそうな場合だけ参照する。

## レビュー手順

1. **対象範囲を確定する**
   - 対象 spec、PR/差分、比較対象ブランチ、レビュー対象外の変更を明示する。
   - 複数 spec にまたがる場合は spec ごとに確認する。
   - 対象 spec 名は `.kiro/specs/<spec名>/` のディレクトリ名から確定し、最終出力では `<spec名>` のようなプレースホルダを残さない。

2. **spec からチェックリストを作る**
   - `requirements.md` の受け入れ基準を、実装確認可能な項目に分解する。
   - `design.md` のインターフェース、データ構造、状態遷移、エラー処理、コマンド、UI 振る舞いを列挙する。
   - `tasks.md` がある場合は、タスクごとに対応する実装ファイルとテストを対応付ける。

3. **差分を中心に確認する**
   - まず PR または `git diff` の差分を読む。
   - 差分だけでは挙動を判断できない場合のみ、関連する既存コード全体を読む。
   - spec にない機能追加、抽象化、挙動変更が混ざっていないか確認する。

4. **実装とテストを照合する**
   - 各受け入れ基準がコードで満たされているか確認する。
   - 重要な分岐、エラーケース、永続化、境界値、UI 状態がテストされているか確認する。
   - テストがない場合は、実装で満たしていても「テスト不足」として報告する。

5. **横断品質を確認する**
   - セキュリティ、入力検証、権限、データ破損、例外処理、ローディング/失敗状態、アクセシビリティ、パフォーマンス劣化を確認する。
   - プロジェクトの既存パターンから外れている実装や、将来の保守を難しくする複雑化がないか確認する。

6. **検証結果を扱う**
   - 依頼や環境に応じて、既存のテスト・lint・型チェックの結果を確認する。
   - 実行できなかった場合は、未実行理由と残るリスクを明記する。
   - コマンド結果だけで判断せず、spec との対応を必ず文章で確認する。

## 確認観点

- **要件カバレッジ**: `requirements.md` の受け入れ基準が実装・テストで満たされているか。
- **設計の遵守**: `design.md` の設計判断・インターフェース・データ構造・状態遷移と実装が一致しているか。
- **タスクの完了**: `tasks.md` の各タスクが実装済みかつ必要なテストを伴っているか。
- **余計な変更の混入**: spec にない仕様変更、UI 変更、リファクタ、依存追加がレビュー対象に混ざっていないか。
- **コード品質と堅牢性**: セキュリティ、入力検証、エラーハンドリング、永続化、並行操作、オフライン動作、境界値に問題がないか。
- **既存仕様との整合**: 他 spec、`docs/requirements-spec.md`、既存のドメインモデルや UI パターンと矛盾しないか。
- **ステアリングとの整合**: `.kiro/steering/development-rules.md` と `docs/CONTRIBUTING.md` の制約を違反していないか。

## 判断基準

- spec と実装が食い違う場合は、原則として `requirements.md` を最優先し、次に `design.md`、次に `tasks.md` を参照する。
- `tasks.md` が `requirements.md` または `design.md` と矛盾する場合は、タスク完了ではなく spec 不整合として報告する。
- 受け入れ基準を満たす実装があっても、回帰しやすい重要挙動にテストがない場合は指摘する。
- spec にない追加実装は、明確に安全で局所的な補助実装でない限り、スコープ逸脱として扱う。
- 重大な不具合、データ破損、セキュリティ、受け入れ基準未達は **High** とする。
- 一部ケースの未対応、テスト不足、設計との差分、既存パターンからの逸脱は **Medium** とする。
- 命名、軽微な重複、説明不足、将来的な改善提案は **Low** とする。

## 出力形式

レビュー結果は次の順で、重大度順に簡潔に出す。

1. **結論**
   - `承認可能` / `修正後に再確認` / `ブロックあり` のいずれかを書く。
   - 対象 PR または差分、対象 spec、確認した範囲を1〜3文で書く。

2. **指摘事項**
   - 問題がある場合は、重大度順に書く。
   - 各指摘には、重大度、該当ファイル、該当箇所、spec 上の根拠、実装上の問題、修正方針を含める。
   - 形式:
     - `High: <短いタイトル>`
       - `場所: path/to/file:line`
       - `根拠: .kiro/specs/<spec名>/requirements.md の <受け入れ基準名>`
       - `問題: <何が満たされていないか>`
       - `修正: <どう直すべきか>`
   - 問題がない場合は「指摘事項なし」と明記する。

3. **spec 対応表**
   - 主要な受け入れ基準と設計項目について、`OK` / `不足` / `不明` / `対象外` を付ける。
   - `不足` と `不明` は必ず理由を書く。

4. **テスト・検証**
   - 確認したテスト、lint、型チェック、手動確認を書く。
   - 未実行の検証があれば理由と残るリスクを書く。

5. **人間の動作確認への申し送り**
   - 人間の動作確認へ進める状態かを書く。
   - `承認可能` の場合でも、即マージ可または spec 完了処理へ進めるとは書かない。
   - 進めない場合は、残タスクを箇条書きで書く。

6. **修正依頼プロンプト**
   - Codex に修正を依頼すべき内容がある場合、そのまま渡せるプロンプトをコードブロックで書く。
   - 修正依頼プロンプト内にコードブロックや Markdown 例を含める場合は、外側のコードフェンスを内側より長くするなど、表示が崩れない形式にする。
   - 修正不要なら「なし」と書く。

7. **ユーザー動作確認案内**
   - レビューの最後に、ユーザーがローカルで最新実装を確認するための再起動コマンドを示し、動作確認を促す。
   - 再起動コマンドはこのスキルに固定せず、`README.md` の開発環境手順と `docs/CONTRIBUTING.md` の Docker 開発・検証環境を確認して、現在の標準手順を示す。
   - 案内文には「上記の手順で再起動したうえで、対象 spec の主要フローを手動確認してください」と明記する。
   - `承認可能` の場合は、Kiro が人間の動作確認待ちで停止すべきことを明記する。

## PR へのコメント投稿

PR 番号または PR URL が指定されている場合は、`/github-operations` を参照してレビュー結果を対象 PR へ投稿する。PR が指定されていないローカル差分レビューでは投稿しない。

- レビュー種別の識別子には `spec-impl` を使う
- 「指摘事項」の各項目を、1 項目につき 1 件、該当ファイルの変更行へのインラインレビューコメントとして投稿する。指摘事項を通常コメントとして投稿してはならない
- 指摘ごとにスレッドを分け、PR 上で返信・解決できる状態にする
- 削除された要素、実装の欠落、複数ファイルにまたがる問題など、問題そのものを表す変更行がない場合も、最も関連する変更行を選んでインラインコメントとして投稿し、コメント本文でその行との関係と指摘対象の範囲を明記する
- 各指摘コメントには、重大度、該当ファイルと箇所、spec 上の根拠、実装上の問題、修正方針を含める
- 指摘投稿後に、結論、指摘件数と重大度、テスト・検証結果、人間の動作確認へ進めるか、未解決事項だけを簡潔にまとめた総括コメントを 1 件、PR の通常コメントとして投稿する
- 指摘事項がない場合は、総括コメントだけを投稿し、「指摘事項なし」と明記する
- spec 対応表の詳細、修正依頼プロンプト、ユーザー動作確認案内は PR コメントに含めず、投稿後のチャット応答にのみ出力する
- 投稿後のチャット応答には、投稿先 PR、投稿したインライン指摘コメント数、総括コメントの投稿結果を記載する
- PR の head commit SHA を取得し、各コメント末尾に次の識別子を入れる
  - 指摘: `<!-- claude-review:spec-impl:<head SHA>:finding:<連番> -->`
  - 総括: `<!-- claude-review:spec-impl:<head SHA>:summary -->`
- 投稿前に既存のインラインレビューコメントと通常コメントを確認し、同じ識別子があるコメントは再投稿しない
- 各投稿の末尾では、`claude-review` 識別子の直前に Claude Code の署名と `<!-- ai-agent:claude-code -->` を入れる

## 指摘対応再確認モード

同じ `/spec-impl-checker` を再度呼び出して実行する。新しいスキルは使わない。

1. `/github-operations` を参照し、PR の最新 head commit SHA、レビューコメント、返信、レビュースレッドの解決状態を取得する
2. `<!-- claude-review:spec-impl:` を含む各未解決指摘について、Codex の返信と署名、返信で示された commit、最新差分を対応付ける
3. 対応した指摘は、修正箇所、関連コード、spec、追加・更新されたテストを確認し、必要な検証を実行する
4. 対応しない旨が返信されている指摘は、その理由を requirements、design、既存コード、テスト結果に照らして独立に評価する。指摘の採用自体を必須としない
5. 各スレッドを次のいずれかに分類する
   - `解決`: 修正が妥当、または対応しない理由が妥当
   - `継続`: 修正不足、回帰、検証不足、または対応しない理由が不十分
   - `未返信`: Codex から判断と根拠の返信がない
6. `解決` のスレッドには再確認結果を返信してから解決する
7. `継続` のスレッドには不足内容を返信し、解決しない。新たに独立した問題を発見した場合だけ、最新 head SHA を使って新しいインライン指摘を投稿する
8. `未返信` のスレッドは解決せず、返信が必要であることを総括に記載する
9. PR の通常コメントへ、解決・継続・未返信の件数、spec 整合性の再判定、検証結果、人間の動作確認へ進めるかを投稿する
10. チャット応答には各指摘の判定を記載する。残る指摘がある場合は Codex へ同じスキルを再度呼び出すプロンプトを出力する。すべて解決した場合は `承認可能` とし、Kiro が人間の動作確認待ちで停止すべきことを出力する。`#spec-completion` プロンプトは出力しない

再確認返信と総括には次の識別子を使い、Claude Code の署名を付ける。

- スレッド返信: `<!-- claude-review:spec-impl:<head SHA>:recheck:<元コメントID> -->`
- 総括: `<!-- claude-review:spec-impl:<head SHA>:recheck-summary -->`

残る指摘がある場合の Codex 向けプロンプト:

```text
$task-implementer 実装 PR #<PR番号> のレビュー指摘に対応してください。
```

## 注意事項

- このスキルは実装レビュー用であり、機能実装を主導しない。
- spec 自体の記述品質レビューが主目的の場合は `/spec-quality-review` を使う。
- spec の要約が必要なだけの場合は `/spec-reader` を使う。
- Kiro はステップ8で spec 完了処理と Issue クローズを行う。Claude Code は、ステップ7の人間の動作確認へ進めるかどうかの判断材料を出す。
- 実装レビューのチャット応答では、`README.md` と `docs/CONTRIBUTING.md` の起動手順を参照して再起動コマンドを示したうえで、ユーザーに手動の動作確認を促す。
- `承認可能` は人間の動作確認へ進めてよいという意味に限定し、即マージ可または spec 完了処理へ進めてよいという意味では扱わない。
