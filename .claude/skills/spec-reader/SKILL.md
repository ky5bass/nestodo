# Kiro spec の読み方ガイド

Kiro spec が長い、複数ファイルに分かれている、または細かな制約を含む場合は、レビューや作業に入る前にこのスキルを使って要点を整理する。

## 読む順番

1. 最初に `.kiro/specs/<feature>/requirements.md` を読む
2. 次に `.kiro/specs/<feature>/design.md` を読む
3. 実装状況やタスク網羅性を確認する場合は `.kiro/specs/<feature>/tasks.md` を読む
4. 全体制約として `.kiro/steering/development-rules.md` を確認する

## 抽出する観点

- `requirements.md` のユーザーストーリーと受け入れ基準
- `design.md` の設計判断、インターフェース、コマンド、データ構造
- `tasks.md` の実装タスクと完了状況
- steering ファイルの制約、特に行数制限と設計理由の明記ルール
- ファイル間の曖昧さ、受け入れ基準の不足、矛盾

## 出力形式

spec を要約するときは、以下の形で簡潔にまとめる:

1. **目的**: その機能が何を達成するためのものか
2. **必須条件**: 受け入れ基準と守るべき制約
3. **設計上の決定**: インターフェース、データ構造、コマンド、アーキテクチャ上の選択
4. **タスク状況**: 計画済み・完了済み・不足している作業
5. **疑問・リスク**: 曖昧さ、矛盾、実装上のリスク

広い言い換えよりも、ファイルパスと具体的な基準を優先する。`requirements.md` と `design.md` が矛盾する場合は、`requirements.md` を正とし、差分を明示する。
