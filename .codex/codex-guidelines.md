# Codex ガイドライン

Codex は作業開始時に必ずこのファイルを読むこと。

思考は英語で行い、最終的な出力は必ず日本語で提供してください。

---

## 実装前の準備（必須）

実装を開始する前に、以下のファイルをこの順で必ず読むこと:

1. `docs/spec-index.md` — spec 一覧、依存関係、関連領域、共通領域
2. `.kiro/specs/<spec名>/tasks.md` — 実装タスク一覧
3. `.kiro/specs/<spec名>/design.md` — インターフェース・データ構造・設計判断の理由
4. `.kiro/specs/<spec名>/requirements.md` — ユーザーストーリーと受け入れ基準
5. `docs/requirements-spec.md` — 全体アーキテクチャ・データモデル
6. `.kiro/steering/development-rules.md` — 共通の開発ルール・spec 運用ルール

`docs/spec-index.md` を確認し、対象 spec に依存 spec がある場合は、実装前に依存 spec の `requirements.md` と `design.md` を読むこと。関連 spec がある場合は、今回の変更が関連領域に触れると判断したときだけ、その spec の `requirements.md` と `design.md` を読むこと。

他 spec の `tasks.md` は、実装順序・未完了タスク・チェックポイント確認が必要な場合のみ読むこと。横断的な依存関係や共通化の判断のために、全 spec の `tasks.md` を毎回読む必要はない。

spec 名が指定されていない場合は、`.kiro/specs/` 配下の候補を確認すること。`_archived` 配下は実装対象から除外し、対象 spec を特定できない場合は実装を開始せずユーザーに確認すること。

ただし、以下のような spec に紐づかない軽作業は、spec 名の特定を必須としない:

- ユーザーが明示した単一または少数ファイルの文書修正（例: `AGENTS.md`, `README.md`, `.kiro/steering/*.md`）
- 実装を伴わない調査、レビュー、方針相談
- 開発環境・設定・CI・ドキュメントの小規模な整備
- ユーザーが具体的な対象ファイルと変更内容を指定した軽微な修正

上記の例外に該当する場合でも、既存 spec の要件や実装方針に影響する可能性がある変更では、関連する spec を確認するか、必要に応じてユーザーに確認すること。

情報が矛盾する場合の優先順位:

1. `.kiro/specs/<spec名>/requirements.md`
2. `.kiro/specs/<spec名>/design.md`
3. `.kiro/specs/<spec名>/tasks.md`
4. 依存 spec の `requirements.md`
5. 依存 spec の `design.md`
6. `docs/requirements-spec.md`
7. `docs/spec-index.md`
8. `AGENTS.md`（この実装ガイド）
9. `.kiro/steering/development-rules.md`

## 実装スコープの原則

- **spec に明記されていない機能・抽象化・最適化は追加しない**
- `tasks.md` のタスクは原則として記載順に進める。ただし `Task Dependency Graph` が定義されている場合は、その依存関係を優先し、同一 wave 内のタスクは実装上自然な順序で進めてよい
- 各タスク完了後に、対応するテストまたは検証を追加・実行する
- ただし、spec 達成に必要な最小限の補助変更（既存コードとの接続、型定義、テストヘルパー、設定調整、今回の変更に起因するバグ修正）は行ってよい。その場合は最終報告に変更理由を明記する
- `tasks.md` で `*` が付いたオプションタスクは、ユーザーが明示した場合、または該当機能の安全な実装・検証に必要な場合のみ実施する。スキップした場合は最終報告に明記する
- 設計に疑問がある場合は `design.md` の「〜という考えから〜とした」箇所を参照する
- 受け入れ基準・データモデル・外部仕様に影響する不明点が残る場合は、実装を止めてユーザーに質問する
- 局所的な実装判断は、既存コードのパターンと spec の意図に合わせて進め、最終報告に判断内容を記載する
- 仕様変更が必要と判断した場合は実装せず、その旨をユーザーに報告する

## プロジェクト構成

技術スタックと全体アーキテクチャは `docs/requirements-spec.md` を正とすること。

## Python 実行環境

- Python / FastAPI 関連のコマンドは、プロジェクト直下の `.venv` を優先して使用すること
- `python` は `.venv/bin/python`、`pip` は `.venv/bin/python -m pip` を使用すること
- `.venv` が存在しない場合は、`.python-version` の Python で `python -m venv .venv` を実行し、`.venv/bin/python -m pip install -e '.[test]'` で依存関係をインストールすること
- グローバル環境や pyenv のベース環境へパッケージをインストールしないこと

---

## 検証コマンド（コミット前に必ず実行）

実際の package script、Docker Compose サービス名、実行ディレクトリを先に確認し、プロジェクトで定義されているコマンドを優先すること。以下はコマンドが未整備の場合の標準候補。

### フロントエンド（Angular）

```bash
npm run build             # 型チェック兼ビルド確認
npm test -- --watch=false # ユニットテスト
npm run lint              # ESLint
```

### バックエンド（Python / FastAPI）

```bash
.venv/bin/python -m pytest      # ユニットテスト
.venv/bin/python -m ruff check . # リント
.venv/bin/python -m mypy .       # 型チェック
```

> 実際のコマンドが判明した場合、または変更された場合はこのファイルを更新すること。検証を実行できない場合は、理由と未検証の範囲を最終報告に明記すること。

## 技術スタック別の規約

### Angular

- スタンドアロンコンポーネントを使用する（`NgModule` は使わない）
- 非同期処理は既存コードの方針（RxJS / Signals）に統一する
- テンプレートの型安全性を維持する（`strictTemplates: true`）

### FastAPI / Python

- 全エンドポイントは `async def` で定義する
- サービス・DB セッションは `Depends` で注入する
- 入出力は Pydantic v2 モデルで定義し、型ヒントを省略しない
- `UpdateInput` パターンでは `model_fields_set` で UNSET と明示的 null を区別する

### SQLAlchemy（async）

- `AsyncSession` を使用する（同期セッション禁止）
- 関連テーブルは `selectinload` / `joinedload` で明示的にロードし N+1 を防ぐ
- 複数テーブルを更新するトランザクションはサービス層で管理する

## エラーハンドリング

- エラーを `@ts-ignore` や `try-catch` で握りつぶさず、根本原因を修正する
- 外部 API・DB 通信は必ず失敗ケースを考慮する
- エラーケースもテストでカバーする
- 今回の変更に起因するエラー、または実装対象に直接関係するエラーは解決してから進む
- 今回の変更と無関係な既存エラーは、原則として修正せず、最終報告に内容と影響範囲を記載する

## コード品質

- 重複を避け、単一の信頼できる情報源を維持する（DRY）
- 意味のある名前で意図を伝える
- 複雑な処理、設計判断、非自明な制約には適宜コメントを挿入する
- コメントは必ず日本語で記述する
- コメントは「なぜ」を書く。「何を」はコードで表現する
- 使用されていないコードは削除する

## テスト

- 実装詳細ではなく振る舞いをテストする
- テスト間の依存を排除し、任意の順序で実行できる状態を保つ
- テストをスキップしない。問題があれば修正する
- エラーケースも必ずカバーする

## セキュリティ

- API キー・パスワードは環境変数で管理する（ハードコード禁止）
- すべての外部入力を検証する
- 必要最小限の権限で動作させる

## Git 運用

- 実装開始前に `git status` で作業ツリーを確認し、Codex の作業対象と既存の未コミット変更を把握する
- ブランチ作成は Codex が判断して行う。現在 `main` で実装・PR 作成が必要な場合は、`feat/<spec名>` を作成して切り替える
- コミットは、`tasks.md` のトップレベルタスク（番号付き）完了時、チェックポイント到達時、spec 完了時、または PR 作成に必要な場合に自動で行う。コミット後、検証結果とコミット内容を報告する
- コンベンショナルコミット形式を使用する
  - フォーマット: `<type>: <説明（日本語）>`
  - 例: `feat: タスク作成APIのバリデーションを追加`
  - 使用可能な type: `build` / `ci` / `docs` / `feat` / `fix` / `perf` / `refactor` / `style` / `test` / `chore`
- コミットは単一の変更に絞る（atomic commit）
- `main` ブランチへの直接コミットはしない
- コミット前に必ず検証コマンドを実行する

## PR 作成

- **PR を作成するタイミング**: `tasks.md` のチェックポイントタスク到達時、または spec の全タスク完了時
  - チェックポイントは「ここまでで動作確認せよ」という設計境界であり、PR の自然な単位となる
  - spec 全体が小規模な場合は 1 spec = 1 PR でよい
- PR 作成の具体的な手順・本文要件・失敗時対応は `.codex/skills/github-pr-creator/SKILL.md` に従う
- 実装 PR には spec ファイルの変更を含めない（spec PR と分ける）
